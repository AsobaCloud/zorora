# SEP-078: Map tile caching for web discovery / imaging views

**Status:** Open
**Created:** 2026-04-03
**Project:** zorora

## Summary
Improve perceived performance and bandwidth use for Leaflet-based maps (imaging / discovery and related views) by caching raster map tiles across sessions and revisits, without breaking attribution or ToS of tile providers.

## Motivation
Today the UI loads basemap tiles from third-party endpoints (for example satellite tiles via WMTS-compatible URLs). Each pan, zoom, and tab re-entry triggers fresh network tile requests. Users on slower links see blank tiles longer; repeat visits do not benefit from durable client-side reuse. Sidebar tab caching (see recent `index.html` work) reduces **application** refetches but does not address **tile** refetches.

## Proposed directions (pick one or combine after spike)
1. **Browser Cache API + service worker** — Intercept tile `GET`s for configured host patterns; store responses with explicit TTL and size budget; fall through to network on miss or expiry. Respect `Cache-Control` from origin when present.
2. **Application tile proxy** — Optional Flask (or CDN) reverse proxy for a subset of zoom/bbox with `Cache-Control` / ETag suitable for browser HTTP cache; keeps API keys or allowlists server-side if ever needed.
3. **Leaflet layer configuration** — Tune `tileLayer` options (`keepBuffer`, `updateWhenIdle`, `maxNativeZoom`) to reduce redundant requests without a new cache layer; low effort, partial win.
4. **Self-hosted or packaged tiles** — For fixed regions (e.g. SADC), evaluate static tile packs or PMTiles for basemap or overlays where licensing permits.

## Constraints
- Must comply with tile provider terms (attribution, hotlinking, rate limits).
- Must not cache authenticated or user-specific responses in a shared cache without review.
- Clear invalidation strategy when basemap URL or style version changes.

## Acceptance Criteria
1. Documented decision: chosen approach (SW, proxy, config-only, or deferred) with rationale and provider constraints.
2. Measurable improvement on at least one scenario: e.g. second load of same viewport shows fewer duplicate tile network requests (DevTools or server logs), or median time-to-interactive tiles improves in a scripted check.
3. No regression: map still loads at default view; attribution text remains visible where required.
4. If a service worker is added: documented scope, update strategy, and how to bypass cache for developers.

## Out of scope (for this SEP)
- Replacing the basemap provider entirely.
- Offline-first full region downloads unless explicitly approved as a follow-up SEP.
- Vector-tile migration (separate design).
