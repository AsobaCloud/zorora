# SEP-080: Complete user-data isolation: pipeline PK, feasibility, legacy greenfield endpoints

**Status:** Open
**Created:** 2026-05-29
**Project:** zorora

## Summary
Finish the auth/data-isolation work started in this branch by closing three remaining gaps so per-user (and team) isolation is actually enforced everywhere.

## Motivation
Gemini implemented research/alerts/scouting-items/user-settings isolation but was cut off before three gaps were closed: (1) upsert_pipeline_asset generates a non-namespaced primary key so two users saving the same source asset collide and overwrite each other via INSERT OR REPLACE; (2) feasibility_results gained user_id/team_id columns but no store method or endpoint uses them and get_feasibility_all is entirely ungated; (3) the legacy greenfield endpoints (/api/scouting/watchlist, /compare, /score) are auth-gated but pass no user_id so they all share the NULL bucket.

## Proposed Change
Namespace the pipeline asset id by user_id (mirroring the watchlist fix); thread user_id/user_ids through the feasibility store methods and gate all feasibility endpoints with ownership checks; wire user_id/user_ids into the legacy greenfield watchlist/compare/score endpoints.

## Acceptance Criteria
Store-level tests prove cross-user isolation for pipeline upsert, feasibility read/write, and watchlist endpoints; an integration test proves the HTTP endpoints reject cross-user access; full modified files compile and existing tests still pass.
