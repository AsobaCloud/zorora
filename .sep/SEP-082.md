# SEP-082: Add user_id migration to research_findings for existing databases

**Status:** Open
**Created:** 2026-05-29
**Project:** zorora

## Summary
The isolation work crashes the app at startup on any pre-existing SQLite database because research_findings.user_id was added to CREATE TABLE and indexed but never migrated onto existing tables.

## Motivation
engine/storage.py _init_schema adds user_id to the research_findings CREATE TABLE and creates idx_user_id on it, but provides no ALTER TABLE ADD COLUMN migration. On a pre-isolation DB, CREATE TABLE IF NOT EXISTS is a no-op so user_id is absent, and CREATE INDEX ... ON research_findings(user_id) raises OperationalError: no such column: user_id, crashing gunicorn at boot (exit 3). This is why ona-zorora-prod:10 crash-loops in Fargate while tests on fresh DBs pass. The alerts and imaging stores already have equivalent ALTER TABLE migrations.

## Proposed Change
Add a guarded 'ALTER TABLE research_findings ADD COLUMN user_id TEXT' (try/except sqlite3.OperationalError) in _init_schema before the idx_user_id index creation, matching the migration pattern in tools/alerts/store.py and tools/imaging/store.py.

## Acceptance Criteria
A LocalStorage initialised against a pre-isolation research_findings table (no user_id) no longer raises, the user_id column and idx_user_id index exist afterward, and a fresh DB is unaffected.
