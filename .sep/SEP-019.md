# SEP-019: Ground Truth Architecture: Deep Research Synthesis Pipeline

**Status:** In Progress
**Created:** 2026-03-03
**Project:** zorora
**Supersedes:** SEP-006, SEP-011, SEP-012, SEP-013, SEP-015

## Summary

Define the target architecture contract for the deep research pipeline, establishing token budgets, data flow contracts, and multi-stage synthesis as the ground truth that all future implementation work must conform to.

## Motivation

The deep research pipeline suffers from three systemic issues that multiple open SEPs address independently, risking piecemeal fixes with vestigial elements:

1. **Context window starvation.** The synthesis prompt consumes ~2620 tokens of the 7B model's ~8192 context, requesting 3072 max_new_tokens for generation. The model silently degrades — regurgitating source content instead of synthesizing — because the prompt's tail (format instructions, credibility scores) falls outside effective attention.

2. **String serialization boundaries.** `academic_search()` and `web_search()` return formatted strings that `parse_academic_results()` and `parse_web_results()` regex-parse back into Source objects. This creates fragile coupling and data loss at every boundary.

3. **Naive quality signals.** `filter_relevant(min_score=0.0)` is a no-op. Relevance scoring uses keyword overlap without stemming or synonyms. Cross-reference detection uses title substring matching. No freshness signal exists.

These issues compound: bad sources pass unfiltered into an overloaded synthesis prompt that the model can't process effectively.

## Architecture Contract

### Principle 1: Token Budget Discipline

**Contract:** No single reasoning model call shall exceed 3000 input tokens.

**Rationale:** The nehanda-v1-7b model has ~8192 token context. With `max_new_tokens=3072` (config), the theoretical input ceiling is ~5120 tokens. However, effective attention degrades above ~3000 input tokens for 7B models, and the Mistral template adds ~50 tokens of overhead.

**Budget table (per-call):**

| Call Type | Max Input Tokens | Max Output Tokens | Total |
|---|---|---|---|
| Subtopic decomposition | 500 | 1024 | 1524 |
| Finding clustering (per-variant) | 2500 | 2048 | 4548 |
| Synthesis outline | 1500 | 1024 | 2524 |
| Synthesis section expansion | 1500 | 1024 | 2524 |

**Enforcement:** All prompt-building functions must calculate char-to-token estimate (÷3.5 for English text) and truncate to stay within budget. The budget constants live in `config.py` under a new `MODEL_BUDGETS` dict keyed by call type.

**Current violations:**
- `_cluster_findings()`: `char_budget=8000` → ~2286 tokens input (borderline OK)
- `synthesize()`: findings + sources + credibility → ~2620 tokens input (exceeds principle when combined with prompt skeleton)
- `format_source_content()`: `_SYNTHESIS_CONTENT_BUDGET=5000` → ~1429 tokens alone

### Principle 2: Multi-Call Synthesis

**Contract:** Synthesis is a two-stage process: outline generation followed by per-section expansion.

**Stage 1 — Outline:**
- Input: refined query + all findings (claims only, no source content)
- Output: 4-6 section titles with 2-3 bullet-point directions per section
- Budget: ~740 tokens input, ~1024 tokens output
- Purpose: structural backbone the model can produce reliably within attention span

**Stage 2 — Section Expansion (4-6 calls, one per section):**
- Input: section title + bullet points from Stage 1 + 2-4 relevant source excerpts + 2-3 relevant findings
- Output: one analytical paragraph (~200-300 words) with inline citations
- Budget: ~820 tokens input, ~1024 tokens output
- Purpose: focused generation with deep attention on a small number of sources

**Assembly:**
- Concatenate executive summary (Stage 1 output preamble) + expanded sections + gaps line
- No post-processing LLM call needed — the structure is built mechanically

**Comparison synthesis:** Same two-stage pattern, but Stage 1 outline uses comparison dimensions instead of themes, and Stage 2 sections compare both subjects on each dimension.

### Principle 3: Structured Data Throughout

**Contract:** Every function in the pipeline operates on typed Python objects (`Source`, `Finding`, `ResearchState`). No function returns formatted strings for another function to parse.

**Current violations:**
- `academic_search()` → returns formatted string → `parse_academic_results()` regex-parses to `List[Source]`
- `web_search()` → returns formatted string → `parse_web_results()` regex-parses to `List[Source]`
- `newsroom_search()` and `brave_news_search()` already return structured data (correct pattern)

**Target signatures:**

```
academic_search(query, max_results) -> List[Source]
web_search(query, max_results)      -> List[Source]
newsroom_search(query, ...)         -> List[Source]  # already correct
brave_news_search(query, ...)       -> List[Source]  # already correct
aggregate_sources(query, ...)       -> List[Source]  # already correct
score_source_credibility(sources)   -> None  # in-place mutation, already correct
score_relevance(query, sources)     -> None  # in-place mutation, already correct
filter_relevant(sources, min)       -> List[Source]
_cluster_findings(query, sources)   -> List[Finding]
synthesize(state)                   -> str  # final output is a string, correct
```

### Principle 4: Centralized Configuration

**Contract:** All numeric limits that affect prompt size, search breadth, or model behavior live in `config.py`. No hardcoded magic numbers in pipeline code.

**New config structure:**

```python
MODEL_BUDGETS = {
    "subtopic_decomposition": {"max_input_chars": 1750, "max_output_tokens": 1024},
    "finding_clustering":     {"max_input_chars": 8750, "max_output_tokens": 2048},
    "synthesis_outline":      {"max_input_chars": 5250, "max_output_tokens": 1024},
    "synthesis_section":      {"max_input_chars": 5250, "max_output_tokens": 1024},
}

SYNTHESIS = {
    "max_findings": 15,           # was hardcoded in format_findings_for_synthesis()
    "max_sources_for_content": 20, # was hardcoded in format_source_content()
    "max_sources_per_section": 4,  # new: limits sources routed to each section expansion
    "max_credibility_sources": 10, # was hardcoded in format_credibility_scores()
    "outline_sections": (4, 6),    # new: min/max sections in outline
}
```

**Char-to-token conversion:** `max_input_chars ÷ 3.5 ≈ max_input_tokens`. Using chars as the enforcement unit avoids needing a tokenizer dependency. The 3.5 divisor is conservative for English text on Mistral-family tokenizers (typical: 3.5-4.0 chars/token).

**Current hardcoded values to migrate:**
- `_SYNTHESIS_CONTENT_BUDGET = 5000` (synthesizer.py:14)
- `char_budget = 8000` (deep_research_service.py:278)
- `sources[:max_sources]` with `max_sources=20` (synthesizer.py:26)
- `state.findings[:15]` (synthesizer.py:20)
- `top_n=10` (synthesizer.py:70)
- `max_results_per_source=3` in preliminary search (deep_research_service.py)

### Principle 5: Source Content Over Metadata in Synthesis

**Contract:** Synthesis prompts maximize source text and minimize structural metadata. Every token spent on URLs, credibility decimals, or redundant headers is a token stolen from the model's ability to reason.

**Current waste (per source in synthesis prompt):**
```
1. [Solid-State Batteries: A Review](https://example.com/article/12345) — Credibility: 0.78
Solid-state batteries use a solid electrolyte instead of liquid...
```
~40 tokens overhead per source for URL + credibility score that the model cannot use for synthesis.

**Target format (per source in synthesis prompt):**
```
[1] Solid-State Batteries: A Review
Solid-state batteries use a solid electrolyte instead of liquid...
```
~8 tokens overhead per source. With 4 sources per section, this saves ~128 tokens — enough for ~45 more words of actual content.

**Credibility's role:** Credibility scores inform *which* sources are selected for the prompt (pre-filtering), not how the model treats them in-prompt. The model cannot meaningfully adjust its reasoning based on `0.78 vs 0.72`.

### Principle 6: Meaningful Quality Filtering

**Contract:** Sources that reach synthesis must pass a relevance threshold > 0.0. The threshold is configurable and defaults to a value that removes the bottom quartile.

**Current state:**
- `filter_relevant(sources, min_score=0.0)` — passes everything
- `score_relevance()` — keyword overlap, no stemming, no synonym expansion
- Cross-references — title substring matching (`title_a in title_b`)
- No freshness signal

**Target state (phased):**

**Phase A (immediate):** Set `min_score=0.15` default. This alone filters ~25% of irrelevant sources based on existing keyword scoring. No new scoring logic needed.

**Phase B (near-term):** Add stemming to relevance scoring via a lightweight stemmer (e.g., Porter stemmer from `nltk.stem` or a 50-line pure-Python implementation). This resolves the current failure where "battery" doesn't match "batteries".

**Phase C (future):** Consider cross-encoder reranking if a small reranker model becomes available on the endpoint. Not in scope for this contract.

## Data Flow: Current vs Target

### Current Pipeline
```
User Query
  → refine_query()         → dict (gaps, dimensions)
  → _generate_query_variants()  → List[str]
  ├─ per variant:
  │   → aggregate_sources()     → List[Source]
  │     ├─ academic_search()    → str → parse → List[Source]  ← VIOLATION P3
  │     ├─ web_search()         → str → parse → List[Source]  ← VIOLATION P3
  │     ├─ newsroom_search()    → List[Source]  ✓
  │     └─ brave_news_search()  → List[Source]  ✓
  │   → score_source_credibility()  → mutates sources  ✓
  │   → score_relevance()           → mutates sources  ✓
  │   → filter_relevant(min=0.0)    → List[Source]  ← VIOLATION P6
  │   → fetch_content_for_sources() → mutates sources  ✓
  └─ merge all sources
  → _cluster_findings(all_sources)    → List[Finding]  ← VIOLATION P1 (borderline)
  → synthesize(state)                 → str            ← VIOLATION P1, P2, P4, P5
```

### Target Pipeline
```
User Query
  → refine_query()         → dict (gaps, dimensions)
  → _generate_query_variants()  → List[str]
  ├─ per variant:
  │   → aggregate_sources()     → List[Source]
  │     ├─ academic_search()    → List[Source]         ✓ P3
  │     ├─ web_search()         → List[Source]         ✓ P3
  │     ├─ newsroom_search()    → List[Source]         ✓
  │     └─ brave_news_search()  → List[Source]         ✓
  │   → score_source_credibility()  → mutates sources  ✓
  │   → score_relevance()           → mutates sources  ✓
  │   → filter_relevant(min=0.15)   → List[Source]     ✓ P6
  │   → fetch_content_for_sources() → mutates sources  ✓
  └─ merge all sources, deduplicate
  → _cluster_findings(per_variant)    → List[Finding]  ✓ P1 (per-variant keeps budget)
  → merge_findings(all_findings)      → List[Finding]  (new: deduplicate cross-variant)
  → synthesize_outline(state)         → OutlineResult   ✓ P1, P2
  → per section:
  │   → route_sources(section, sources) → List[Source]  (new: select relevant sources)
  │   → synthesize_section(section, sources, findings) → str  ✓ P1, P2, P5
  → assemble_synthesis(outline, sections) → str         ✓ P2
```

## Implementation Phases

### Phase 1: Token Budget Foundation (SEP-013 scope)
**Files:** `config.py`, `workflows/deep_research/synthesizer.py`
- Add `MODEL_BUDGETS` and `SYNTHESIS` dicts to config.py
- Replace all hardcoded limits in synthesizer.py with config references
- Set `filter_relevant` default to 0.15
- Drop URLs and credibility decimals from synthesis source format (P5)
- **Does not** change the single-call synthesis pattern yet — just gets the budget right

### Phase 2: Multi-Stage Synthesis (SEP-006/SEP-011 scope)
**Files:** `workflows/deep_research/synthesizer.py`, `engine/deep_research_service.py`
- Split `synthesize()` into `synthesize_outline()` + `synthesize_section()` + `assemble_synthesis()`
- Add `route_sources()` to select per-section relevant sources
- Update `run_deep_research()` to call the new multi-stage pipeline
- Remove old single-call synthesis code (no vestigial elements)

### Phase 3: Structured Data Flow (SEP-012 scope)
**Files:** `workflows/deep_research/aggregator.py`, search provider functions
- Refactor `academic_search()` and `web_search()` to return `List[Source]` directly
- Remove `parse_academic_results()` and `parse_web_results()` regex parsers
- Remove string-formatting code from search functions

### Phase 4: Quality Signals (SEP-015 scope)
**Files:** `workflows/deep_research/reranker.py`, `workflows/deep_research/credibility.py`
- Add stemming to relevance scoring
- Replace title substring cross-reference detection with source_id based matching
- Prefer `content_full` over `content_snippet` in finding extraction
- Add freshness signal (publication_date → recency bonus)
